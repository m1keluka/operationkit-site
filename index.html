An honest pre-release note — read this first.

The core runs up to ~25 agents executing shell on a single shared host. That is a real blast radius, and we say so plainly. Current isolation is per-session git worktrees plus a pre-write guard hook — enough to stop the common cross-session footgun, not a security sandbox.

Per-agent container/sandbox isolation is on the roadmap, not shipped. Until then, run OperationKit on a dedicated, disposable VPS with only the credentials it needs. We document the sharp edges instead of hiding them — that honesty is part of the brand.

A fleet, not a tab

Built to govern many agents at once — not babysit one.

Most tools manage one agent, on one machine, for one person. OperationKit is the control plane for a whole fleet. Four things make that real, in this order.

01

Self-hosted by design — your infra, your cost, your data.

OperationKit runs entirely on a server you control: a $6 VPS, your own Anthropic billing, your own data-residency posture. Nothing routes through a third-party orchestration cloud. This is the one thing a vendor cloud structurally cannot offer — so it leads.

02

A fleet, not a tab — multi-tenant workspaces.

Built for 25+ concurrent sessions organized into multi-tenant workspaces — separate teams or clients on one install, each with its own objectives and members, without crossing the streams.

03

Governance is built in, not bolted on.

Role-based access control and per-workspace personas mean the right people get the right agents with the right context. It's the layer teams and agencies actually pay attention to — and the layer single-user tools skip.

04

An adversarial review gate, not just a diff.

Work produced by an agent can be routed through an automated adversarial review before it's accepted. Most tools stop at "show the human a diff"; OperationKit treats review as a governed stage of the workflow. The board of objectives is one feature inside that control plane — not the product.

Why not just use a hosted Claude Code product?

Hosted Claude Code is excellent — use it if it fits. OperationKit exists for what a vendor cloud structurally can't serve:

Your code and credentials never leave your infrastructure. Everything runs on a host you control. Nothing is sent to a third-party orchestration cloud.

Multi-tenant governance. Workspaces, RBAC, per-workspace personas, and a review gate — built for a team or agency running many projects, not a single user.

Your infra, your cost, your rules. One cheap VPS, your own billing, your own data-residency and compliance posture.

If you're a solo dev happy on a vendor cloud, the hosted product is probably the better choice. OperationKit is for keeping the whole operation in-house.

From zero to fleet

Stand it up on a $6 VPS in minutes.

One host, one compose file, then drive the board from your laptop or your phone. The snippet is illustrative — the real steps live in the self-hosting guide.

1

Bring a host.

Any disposable VPS — a $6/mo box is enough to start. Add your own Anthropic API key or org seats.

2

Compose up.

Clone, set env, docker compose up. The web app and the agent runtime come up together.

3

Open the board — anywhere.

Create a workspace, spawn objectives, watch each live session in an embedded terminal from your phone.

ssh root@your-vps · ~/operationkit
# on your own $6 VPS — nothing leaves this host $ git clone https://github.com/m1keluka/OperationKit.git $ cd OperationKit && cp .env.example .env # add your own ANTHROPIC_API_KEY in .env, then: $ docker compose up -d ✓ web ready → http://localhost:3000 ✓ runtime ready fleet: 0/25 sessions ✓ review armed adversarial gate enabled # open the board from your laptop or your phone.
